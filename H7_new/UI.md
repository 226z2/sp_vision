# DM-balanceV1 UI/裁判叠加任务指南 (H7_new)

说明 UI 生成与裁判串口传输的数据流，便于快速定位与扩展。

## 主要文件与职责
- 任务：`User/APP/UI.[ch]`
  - 周期构建 UI 图元列表并通过裁判串口发送。
- 绘制库：`User/ui/` 与 `User/ui_back/`
  - `ui_interface.[ch]`, `ui_types.h`, `ui_g.[ch]` 定义图元/协议。
- 数据来源：`User/APP/referee.[ch]`（裁判帧解析）、`referee_usart_task.[ch]`（串口收发）、运行时状态（模式/功率/弹丸/姿态/自瞄）。

## 数据流
1) 输入
   - 裁判数据：血量、功率、热量、比赛状态。
   - 机器人状态：底盘/云台/射击模式与目标，自瞄锁定，电容/功率，错误检测。
   - 可能的外部标志：通信/自瞄结果。

2) 组包流程（UI 任务）
   - 挑选需要显示的字段（HP、模式、弹速/热量、功率、自瞄状态）。
   - 调用 `ui_interface` 生成图元（文字、线框、条形图等），写入发送缓冲。
   - 控制发送节奏：裁判带宽有限，按帧或分组下发，避免刷新过快。

3) 发送
   - 使用裁判串口（`referee_usart_task`）按协议发送图元更新。
   - 失败/超时可重试或减小刷新频率。

4) 输出
   - 客户端叠加界面显示实时信息，辅助驾驶与调试。

## 典型链路
- referee 串口接收 → `referee` 解析 → UI 取状态 → `ui_interface/ui_g` 组图元 → 裁判串口发送 → 客户端显示。

## 关键配置
- 图元 ID/层级/颜色：在 `ui_g`/`ui_interface` 里定义，需保持唯一。
- 刷新频率与批量大小：UI 任务内部控制，需兼顾带宽。
- 显示内容：可在 UI 任务中自定义字段组合。

## 调试建议
- 初次调试时减少图元数量，确认通路无 CRC/帧格式错误。
- 逐步增加刷新频率，监控裁判带宽是否足够；必要时分帧发送。
- UI 问题优先排查裁判串口收发与协议字段，而非上层逻辑。
